# 東京レイルコンフォート（仮）— 構想・設計書

> Claude Code 実装用のプロジェクトブリーフ。リポジトリ直下に `docs/CONCEPT.md` として置くか、`CLAUDE.md` のベースとして使う想定。

## 0. 一行でいうと

東京の電車の混雑を「統計 × 時刻表」で時間帯別に推定し、リアルタイムの運行情報（遅延）で補正して可視化。さらに「空いている時間・区間」を提案して、オフピーク移動＝手取り時間の増加を後押しする Web アプリ。

**都知事杯オープンデータ・ハッカソン2026 個人参加作品**

## 1. 大前提：データの現実（ここを外すと詰む）

### ❌ 取れないもの
- **リアルタイムの車両別・列車別の混雑度（実測）は、東京の電車ではオープンデータに存在しない。**
  - 東京メトロの ODPT リアルタイムデータは「運行情報（遅延・運休）」と GTFS-RT の「Alert（運行アラート）」のみ。混雑は含まれない。
  - 都営地下鉄も GTFS-RT は Alert のみ。
  - JR 東日本の GTFS-RT は列車位置・遅延を配信するが、混雑（occupancy）フィールドは入っていない。
  - 各社の自社アプリ（my! Tokyo Metro 等）が持つ車両別混雑は非公開。
- **→ 実装方針：「実測」を諦め、「推定（予測）」＋「リアルタイム補正」で構成する。UI 上も "推定" と明示する（誠実さが審査でも効く）。**

### ✅ 取れるもの（＝土台にするデータ）
| データ | 内容 | 粒度/更新 | 出所 |
|---|---|---|---|
| 国交省 都市鉄道混雑率調査 | 東京圏 主要31区間の混雑率 | 路線×区間、朝ピーク1時間、年1回、PDF | 国土交通省 |
| 駅別乗降人員 | 各駅の1日平均乗降人数 | 駅単位、年1回 | 各鉄道事業者／東京都オープンデータカタログ |
| 駅時刻表 / 列車時刻表 | 何時に何本走るか | 時間帯別、静的 | ODPT（odpt:StationTimetable / odpt:TrainTimetable） |
| 運行情報 | 遅延・運休・見合わせ | リアルタイム | ODPT（odpt:TrainInformation） |
| 列車ロケーション | 列車の在線位置 | リアルタイム（一部事業者） | ODPT（odpt:Train） |
| 駅・路線マスタ | 駅座標、路線、駅順 | 静的 | ODPT（odpt:Station / odpt:Railway） |

## 2. データソース詳細（実装リファレンス）

### 2-1. ODPT（公共交通オープンデータセンター）
- ベース URL: `https://api.odpt.org/api/v4/`
- 認証: 無料のユーザー登録で `consumerKey`（アクセストークン）を取得。各リクエストに `?acl:consumerKey=<KEY>` を付与。
- 形式: JSON（一部 GTFS/GTFS-RT）。
- 主なエンドポイント（例）:
  - 運行情報: `odpt:TrainInformation?odpt:operator=odpt.Operator:TokyoMetro`
  - 列車ロケーション: `odpt:Train?odpt:operator=odpt.Operator:TokyoMetro`
  - 駅時刻表: `odpt:StationTimetable`
  - 列車時刻表: `odpt:TrainTimetable`
  - 駅情報: `odpt:Station`
  - 路線情報: `odpt:Railway`
- 事業者コード例: `odpt.Operator:TokyoMetro`（東京メトロ）, `odpt.Operator:Toei`（都営）, `odpt.Operator:JR-East`（JR東日本）。
- **注意**: 通常の ODPT センターに加え、「公共交通オープンデータチャレンジ」期間限定で提供が拡充されるデータ（京王・小田急・京急・相鉄など）がある。ライセンスが「チャレンジ限定」なので、都知事杯での利用可否・期間はカタログの各データのライセンス表記を必ず確認する。まずは通年提供の東京メトロ・都営・JR東日本で組む。

### 2-2. 国交省 混雑率データ
- 「都市鉄道の混雑率調査結果」。最新は令和6年度実績（2024年データ、2025年7月公表）。
- 資料2「三大都市圏の主要区間の混雑率」に、事業者名・線名・区間・時間帯・輸送力・輸送人員・混雑率が載る（東京圏31区間）。
- **形式は PDF。構造化データ化（CSV 変換）が最初の作業。** Claude Code に PDF をパースさせて `congestion_baseline` テーブルに投入する。過去年分も取れば経年比較も可能。

### 2-3. 駅別乗降人員
- 各社サイト or 東京都オープンデータカタログ（`https://portal.data.metro.tokyo.lg.jp/`）から取得。
- 駅ごとの相対的な重み付け（混雑推定の按分）に使う。

### 2-4. ライセンス遵守（提出要件にも直結）
- **リアルタイムデータ（運行情報・列車位置）は、データ生成時刻 `dc:date` を画面に表示する。**
- **「本アプリは公共交通オープンデータセンターのデータを利用しています」等の出典表示を入れる。**
- 静的データ（時刻表・駅情報）はキャッシュ（DB 保存）可。混雑率・乗降人員も静的マスタとして保持してよい。
- 各データの取得日時・出典・更新頻度を明示（古い情報による誤解防止のガイドライン要件）。

## 3. 混雑推定モデル（MVP はシンプルに）

```
推定混雑指数(区間, 時間帯, 日種別)
  = ベース混雑率(区間)                       # 国交省データ（ピーク=100%基準）
    × 時間帯係数(区間 or 路線, 時)            # 時刻表の本数 + 一般的な需要カーブで按分
    × 曜日係数(平日/土休日)                  # 平日ピークを1.0, 休日を下げる 等
    × リアルタイム補正(遅延分)               # 遅延min→混雑上昇の簡易係数
```

- 出力は 0–1 or % に正規化し、**4〜5段階（快適／やや混雑／混雑／かなり混雑／激しい混雑）** にマッピングして色で表示。
- 「実測ではなく推定です」の但し書きを常時表示。
- MVP は**係数を素朴なルールベース**で置く（機械学習は後回し）。時間帯カーブは「朝7-9・夕17-19がピーク」の一般形＋時刻表本数で十分。
- 発展形：ユーザー投稿（クラウドソース）を事後の補正項として足す（下記 5. の Phase 2）。

## 4. アーキテクチャ（吉村さんの既存スタックに合わせる）

```
[GitHub Actions（定期バッチ）]
  ├─ 混雑率PDF → パース → congestion_baseline へ（初回 + 年次）
  ├─ 駅別乗降人員 → station_master の weight 更新（初回 + 年次）
  └─ ODPT 時刻表 → 時間帯別本数を集計 → timeband_factor へ（週次など）
        │
        ▼
[Supabase (Postgres)]  ← 静的マスタ + 推定テーブルを保持
        │
        ▼
[Next.js + TypeScript + Tailwind (on Vercel)]
  ├─ Server Component / Route Handler / Edge Function
  │     └─ ODPT 運行情報をリアルタイム取得（短時間キャッシュ）→ 補正
  ├─ 混雑推定を算出（DBの静的値 × リアルタイム補正）
  └─ UI: 路線図/区間リストの混雑ヒートマップ + 時間帯グラフ + 空き時間提案
```

- **リアルタイム取得**は Route Handler / Edge Function 経由にして `consumerKey` をサーバ側に隠す（クライアントに出さない）。レスポンスは 30〜60秒キャッシュで API 負荷を抑える。
- **静的処理**は GitHub Actions でバッチ化し Supabase に流し込む（既存の GitHub Actions 運用ノウハウがそのまま活きる）。
- デプロイは Vercel。特典の Cloudflare 環境を使う場合はデモ公開先として選択可。

### DB スキーマ（初期案）
```sql
-- 路線マスタ
railways(id, operator, name_ja, name_en, color, order_index)

-- 駅マスタ（座標・乗降人員weight）
stations(id, railway_id, name_ja, name_en, lat, lng, order_index, daily_passengers, weight)

-- 区間（駅間）マスタ
segments(id, railway_id, from_station_id, to_station_id, order_index)

-- 混雑ベースライン（国交省）
congestion_baseline(id, segment_id, direction, peak_time_band, capacity, ridership, congestion_rate, fiscal_year, source_url)

-- 時間帯係数（時刻表由来）
timeband_factor(id, railway_id, day_type, hour, train_count, factor)

-- （Phase2）ユーザー投稿
user_reports(id, segment_id, reported_at, level, source, lat, lng)
```

## 5. スコープ：MVP → 発展

### Phase 0（MVP・ハッカソン提出の核）
- 対象を**混雑上位＆分かりやすい2〜3路線**に絞る（例：JR山手線・中央快速線・東京メトロ東西線）。全路線は狙わない。
- 機能：
  1. 区間ごとの**時間帯別 推定混雑**を色で可視化（一覧 or 簡易路線図）。
  2. 「今」の推定混雑＋リアルタイム運行情報（遅延）の反映。
  3. **「空いている時間帯の提案」**（この区間、8時台は激混みだが10時台は快適、等）を1機能。
- 出典表示・`dc:date` 表示・「推定です」注記を実装（ライセンス遵守＝加点）。

### Phase 1（Final Stage までに余裕があれば）
- 地図（MapLibre 等）に路線を重ね、混雑ヒートマップ表示。
- 出発駅→目的地で「空いている時間に乗るなら何時」を提案するミニ経路提案。

### Phase 2（Demo Day / 実装支援フェーズ向け）
- **ユーザー投稿型の混雑報告**（Google マップ的）で推定を補正。データ生成ループが回る＝"サービス"として強くなる。
- 夏の暑さデータ（気温・駅の地上/地下）と掛けて「空いていて涼しい移動」提案（当初の避暑アイデアの合流点）。

### 非ゴール（やらないことを明示）
- リアルタイム車両別混雑の"実測"表示（データが無い）。
- 全事業者・全路線の網羅（MVP では割り切る）。
- 高度な機械学習モデル（MVP はルールベース）。

## 6. ハッカソン上の位置づけ

- **狙う賞**：データ活用賞（複数オープンデータの掛け合わせ）／ソーシャルインパクト賞（手取り時間・QOL、国交省のオフピーク通勤推進と一致）／サービスデザイン賞（体験設計）。
- **プレゼン（2分動画）の骨子**：
  1. 課題：東京の通勤混雑は QOL を下げるが、"今どこが空いているか"を事前に選べない。
  2. でもリアルタイム混雑はオープンデータに無い → **統計×時刻表×運行情報で"推定"するアプローチ**が新規性。
  3. デモ：区間の時間帯別混雑と「空き時間提案」。
  4. インパクト：オフピーク移動の後押し＝国の政策目標とも一致、都民の手取り時間を増やす。
- **エントリー項目（各300字）への布石**：使用オープンデータ（ODPT 運行情報・時刻表・駅情報／国交省混雑率／駅別乗降人員）を代表10件として整理しておく。

## 7. Claude Code への最初の指示（叩き台）

1. Next.js (App Router) + TypeScript + Tailwind + Supabase クライアントの雛形を作成。
2. `docs/` にこの構想を配置し、`stations / railways / segments / congestion_baseline / timeband_factor` のスキーマで Supabase マイグレーションを生成。
3. ODPT の `TrainInformation` を叩く Route Handler（`consumerKey` は env、レスポンスキャッシュ付き）を実装。
4. 国交省 混雑率 PDF（サンプル1年分）をパースして `congestion_baseline` に投入するスクリプトを作成。
5. 対象2路線分の推定混雑を計算して色分け表示する最小 UI を作成。まずはモックデータでも可、次に実データに差し替え。

> 実装を始める前に、ODPT 開発者サイトで `consumerKey` を取得し、`.env.local` に置くこと。国交省 混雑率 PDF は最新年度分をダウンロードして `data/` に置く。
