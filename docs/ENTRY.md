## ② サービス概要 (課題・背景) — 約270字

東京の通勤電車の混雑は都民のQOLと生産性を下げ続けているが 「今どの区間がどれだけ混むか」を事前に比べて行動を変える手段がない 車両単位のリアルタイム混雑の実測値はオープンデータに存在せず 各社アプリの混雑情報も路線横断では比較できない そこで本作品は 国土交通省の混雑率統計とODPTの時刻表から時間帯別の混雑を推定し リアルタイム運行情報 (遅延) で補正して路線横断で可視化する さらに「この区間 何時なら空いているか」を提案し オフピーク移動による快適な移動と手取り時間の増加を後押しする 国のオフピーク通勤推進とも方向が一致する

## ③ プロダクト・技術詳細 — 約280字

「混雑の実測データが存在しない」という制約を 統計×時刻表×リアルタイム補正の3層推定で乗り越えるのが技術的特徴 国交省 都市鉄道混雑率調査の区間別ピーク混雑率をベースに ODPT時刻表から算出した時間帯係数と曜日係数を掛け 運行情報の遅延分を補正項として反映する 推定ロジックは副作用のない純関数として実装し単体テストで担保 UIには「実測ではなく推定」の注記とデータ生成時刻・出典を常時表示し オープンデータ利用ガイドラインを遵守する 技術構成は Next.js + TypeScript + Supabase + Vercel GitHub Actions で静的データを定期取込みし 遅延履歴の蓄積による推定精度の継続改善も設計済み

## ④ チーム実行力 — 約230字 (※実績部分は本人が具体名を入れる)

個人参加 Web開発 (Next.js/TypeScript) とデータ処理 (Python) の両方を扱い 個人開発でWebサービスの企画から実装・公開・運用までを完結してきた実績がある (例: 本番稼働中の◯◯ ◯◯ など) 本プロジェクトは既に Next.js + Supabase の土台構築と混雑推定エンジンのプロトタイプ実装まで完了しており エントリー時点で動くモックが存在する ハッカソン期間は GitHub Actions による自動化と生成AIコーディング支援を活用し 短期間で MVP からデモ公開まで到達する
